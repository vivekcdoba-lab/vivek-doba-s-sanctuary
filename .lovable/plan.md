# Agreement: pristine template + B1.1/B1.2 before signature, "Open Agreement" only after signing

## What's wrong today

1. The current code (`buildClientPages.ts` + edge functions) **prepends** B1.1 / B1.2 at the very front of the PDF the seeker sees — that's the wrong position and visually changes the uploaded template's order.
2. The "Open Agreement →" link in `SeekerDetailPage.tsx` Documents tab is **always visible** and points to the in-app React renderer (`/seekers/:id/premium-agreement`), not to the actual signed PDF. The user wants it hidden until the document is signed, and then it should open the real signed PDF (template + B1.1 + B1.2 + signature certificate).
3. The signed PDF generated by `submit-signature` and `sign-document-inline` currently puts B1.1 / B1.2 at the front (wrong) and the signature certificate at the back. Position must be: original template pages → B1.1 → B1.2 → signature certificate (last).

## Fix — three coordinated changes

### 1. Insert B1.1/B1.2 just before the signature page, never at the front

Update `supabase/functions/_shared/buildClientPages.ts`:

- Rename the exported helper from `prependClientPages` to `appendClientPages` (or add a second function — keep one to avoid drift).
- Replace `pdfDoc.insertPage(0, ...)` and `pdfDoc.insertPage(1, ...)` with `pdfDoc.addPage([595, 842])` so both pages are added **at the end** of the original template — without touching, re-ordering or re-rendering any of the uploaded template's existing pages. The uploaded PDF's bytes stay byte-for-byte identical except for two new pages tacked on.
- Footer page numbers ("1 | Page", "2 | Page") are misleading once these pages are no longer at the front — change them to `"B1.1"` and `"B1.2"` chips so they self-identify regardless of the underlying page count.

### 2. Update both edge functions to call the appended variant in the new order

`supabase/functions/get-signature-request/index.ts` (preview the seeker sees before signing):

- Call `appendClientPages(pdfDoc, ...)` instead of `prependClientPages(...)` so the in-browser preview matches what they will sign: original template pages first, then B1.1, then B1.2.
- Continue uploading to `documents/_preview/{request_id}.pdf` and returning a 30-min signed URL. (No DB or schema change.)

`supabase/functions/submit-signature/index.ts` (final signed file emailed + stored):

- Order of operations becomes:
  1. Load the original template bytes — **don't modify them**.
  2. `await appendClientPages(pdfDoc, { seeker, fee })` → adds B1.1 + B1.2 at the end.
  3. `pdfDoc.addPage([595, 842])` for the existing signature certificate page.
- Net result: pages are `[…template…, B1.1, B1.2, Signature Certificate]`. For the standard 10-page agreement template that produces B1.1 = page 11, B1.2 = page 12, signature = page 13, exactly matching the user's "pages 11 and 12" requirement.

`supabase/functions/sign-document-inline/index.ts` (in-person "Sign Here" flow):

- Same change: load template untouched → append B1.1/B1.2 (we'll need to fetch `seeker` profile + latest `agreements` row of `type='fee_structure'` here too — currently this function only knows the doc title) → append signature certificate page. This keeps in-person and email flows producing identical artifacts.

### 3. Gate "Open Agreement" behind a signed Premium Agreement and point it to the signed PDF

In `src/pages/admin/SeekerDetailPage.tsx` (the Documents tab block around lines 837–854):

- Query the seeker's signature requests for any **signed** request whose document is the Premium Coaching Agreement (match by `documents.title ilike '%premium%coaching%agreement%'` OR by `documents.category = 'agreement'` — confirm which is reliable; today the SeekerSignaturesTab already loads this exact list, so we can lift it via a small hook `useSeekerAgreementSignedPdf(seekerId)` that returns `{ signedPath, signedAt } | null`).
- Render rules in the "Premium Coaching Agreement" card:
  - **Not yet signed** → hide the "Open Agreement →" button entirely. Show a small muted line: *"Will become available once the seeker signs the agreement."*
  - **Signed** → replace the button with **"Open Signed Agreement →"** that calls `supabase.storage.from("signatures").createSignedUrl(signedPath, 60)` and opens the URL in a new tab. This is exactly the same mechanism `SeekerSignaturesTab` already uses for its Download button — so the user gets the real signed PDF (template + B1.1 + B1.2 + signature certificate), not the React preview.
- The internal preview route `/seekers/:id/premium-agreement` is **not removed** (preservation policy) — admins can still reach it directly if needed; it's just no longer exposed as the default action from the Documents tab.

Apply the identical change in `src/pages/coaching/CoachSeekerDetail.tsx` if it exposes the same "Open Agreement" link (will verify and mirror).

## Files touched

- `supabase/functions/_shared/buildClientPages.ts` — switch insert→add (append at end), update page-number footer chips.
- `supabase/functions/get-signature-request/index.ts` — call append variant.
- `supabase/functions/submit-signature/index.ts` — call append variant; order = template → B1.1 → B1.2 → certificate.
- `supabase/functions/sign-document-inline/index.ts` — same; also fetch seeker + fee_structure to feed the helper.
- `src/pages/admin/SeekerDetailPage.tsx` — gate "Open Agreement" on signed status; switch link target to signed PDF.
- `src/pages/coaching/CoachSeekerDetail.tsx` — mirror gating if applicable.

No DB migration, no schema change, no new buckets, no new secrets. Existing `signatures` bucket and `document_signatures.signed_pdf_path` already hold what we need.

## How to verify after deploy

1. Admin creates / confirms a Fee Structure for the test seeker.
2. Documents tab: confirm the "Open Agreement →" button is **hidden** and the muted "will become available once signed" hint is shown.
3. Click **Send Email for Signature** → open the email link in a private window. The preview iframe shows the original template untouched, followed by B1.1 (Client Details) and B1.2 (Payments & Fees) at the end, just before the seeker submits the signature form.
4. Submit the signature. Re-open the seeker's Documents tab — the button now reads **"Open Signed Agreement →"**, opens the file from the `signatures` bucket, and the PDF order is: original template pages → B1.1 → B1.2 → Digital Signature Certificate.
5. Repeat with the in-person **Sign Here** flow → the resulting signed PDF must have the same page order as the email flow.
